These are the mining procedures for every source the skill covers. **Apply only the section(s) for your assigned stream's source(s)** — mine that source to its definition of done and ignore the procedures for other sources. Prefer current, on-main evidence; treat historical/chat/session evidence as supporting. **Thoroughness is the whole point** — never sample a source, read only a preview, or silently skip it. Record the exact searches/queries/IDs you ran, and any gap (`partial`/`no-access` with the exact blocker), in your coverage note.

> **The external-source procedures (items 9–12) name examples from one company's toolchain — Slack, WorkIQ/M365, the Kusto/Trino warehouse.** Treat each as a _capability_ (team chat, shared docs/email, telemetry), not a fixed vendor: a team on a different stack (e.g. Google Workspace — Gmail, Google Docs/Drive, Google Chat, Meet) has the same categories with different tools. Use whichever connector is actually available for the target org; mine what you can reach (e.g. public/exported doc URLs via `markitdown`). If the org's connector for a capability is absent from your context, follow the connector-source rule below — do not silently skip the category.

> **Connector-source rule (items 2 and 9–12).** These sources need a tool — `session_store_sql`, team-chat (Slack/…), shared-docs/knowledge (WorkIQ/…), telemetry/warehouse, or a doc-parser (`markitdown`/…) — that **may not have been surfaced to your context** (tool availability is non-deterministic across agents). Each connector stream has **one primary connector** (the capability named in your stream context); a few procedures also use a **secondary tool** (e.g. a doc-parser for URLs the primary surfaces — see item 10). So before mining one, **assert its connector is callable as your very first action** (e.g. run a tiny `session_store_sql` count; confirm the chat/knowledge/warehouse/doc-parser tool responds). **If the required _primary_ connector is absent, STOP: emit `TOOL_MISSING: <stream-id> <capability>` as your output, write no candidate file, and do _not_ record `no-access` or fabricate findings.** A missing connector is a dispatch problem for the orchestrator to resolve (it will retry on a fresh agent, or mine the stream itself as a last resort) — it is not a coverage outcome you may decide. **A _secondary_ tool that is missing must _not_ stop the stream:** mine everything the primary yields, then record the secondary as a **per-tool `partial`** in your coverage note — name the missing tool and exactly what went uncovered (e.g. the specific doc URLs you could not parse), so the orchestrator can follow up via the stream that owns that tool. Record `no-access` **only** when the connector _is_ present but a specific resource needs a permission/scope/credential you lack (name the exact access). The `gh`/GraphQL artifact sources (items 3–8) are not connector-backed and need no such assertion.

1. **Main branch — read for signal, prioritized by impact.** Always fully read the high-signal roots: `README`, `CONTRIBUTING`, `.github/copilot-instructions.md` / `AGENTS.md`, all CI workflows under `.github/workflows/`, `.github/CODEOWNERS`, build manifests (`package.json`, `go.mod`, `Cargo.toml`, …), every build/release script (`script/`, `Makefile`), and `docs/`. Then cover **every top-level subsystem** of the source tree — but the bar for a subsystem is **signal-saturation, not literally every file** (see the code-completeness bar). Large UI/leaf-component trees (e.g. hundreds of small components) rarely yield durable memories; prioritize **entry points, config, public interfaces, README/docs in the subdir, and the hot-spot files** the hot-file ranking (git-churn-led, with session edit-counts as a supporting hint, provided in your task context) points to. **When your task context names a `.files` read-set** (CODE/TEST streams), that file list is your stream's exact, hot-ordered assignment — read it to the saturation bar and do **not** re-enumerate the tree yourself; the high-signal roots and hot-spot prioritization above apply **within** it (across the fleet, every mineable file lands in exactly one stream's read-set, so a root file you don't see belongs to another stream).
2. **Cloud agent sessions** — `session_store_sql`, scoped to this repo (a **connector-backed** source — assert it first, per the connector-source rule above). Mine for what agents repeatedly searched for, got stuck on, or got wrong — recurring searches, repeated stuck points, and corrections (from `turns`/`events`). It is a **timeout trap**: never join the big tables to `sessions`, never `SELECT *` from the wide `events` table, and always time-filter, `LIMIT`, and select few columns. The reliable pattern is **fetch this repo's recent session IDs first, then read details in `session_id IN (…)` chunks** (≤~40 ids for `turns`, ≤~15 for the wide `events` table):

   ```sql
   -- 1) recent sessions for this repo (cheap: exact-match + time filter). `repository` is the
   --    session's repo and CAN be NULL, so this may undercount.
   SELECT id FROM sessions
   WHERE repository = '<owner>/<repo>' AND created_at > now() - INTERVAL '6 months'
   ORDER BY created_at DESC LIMIT 200;
   -- 2) then read details by session-id chunk — NEVER `session_files`/`turns`/`events` ⋈ `sessions`.
   --    stuck points / recurring searches / mistakes (chunk `turns` ≤~40; `events` ≤~15, never SELECT *):
   SELECT session_id, turn_index, user_message FROM turns
   WHERE session_id IN ('<id1>', '<id2>', …) LIMIT 500;
   ```

3. **GitHub Actions** — Use the GitHub MCP server to retrieve failing workflow runs across a broad recent window; identify the most common **non-flaky** failure causes. Also read every workflow file to know what each gate enforces. **Exclude non-failures masquerading as failed runs** — e.g. `notify-pending-deployment` / deploy-approval gates whose logs are absent (`log not found`); these are approval waits, not CI breakages, and are noise.
4. **Pull requests** — Use the GitHub MCP server to retrieve all pull requests and comments. Mine **review threads** at scale for recurring reviewer feedback; mine merged/reverted PRs for churn and rollback patterns. Sweep dozens or hundreds, not a handful.
5. **Issues** — Use the GitHub MCP server to retrieve all relevant issues, open or closed. **Don't assume a single issues repo.** Issues about this repo may live in the repo itself, in one or more _other_ repos (a dedicated planning/tracking repo, an umbrella/monorepo, a sibling repo, …), or a mix — **identify every repo that hosts issues about this one and mine each** (run the command above per repo; your task context may already list them). Find them from the code repo's own issues, including _transferred_ or cross-referenced ones (`org/repo#123`); its `.github/ISSUE_TEMPLATE/config.yml` contact links, docs, CONTRIBUTING, and escalation metadata; the repos its PRs close issues in; or any other signal you can find. Treat the code repo itself as a candidate unless its Issues tab is disabled or explicitly redirects elsewhere. **When an issues repo also tracks other projects** (a shared tracker/monorepo), scope to issues about _this_ repo — filter by area/component label or title/body/cross-reference mentions — and don't mine the whole tracker. Record any issues repo you suspect exists but cannot locate or reach as a gap in your coverage note.

10. **Shared docs / email / meeting transcripts / retros** — for example via **WorkIQ** (`workiq-ask`), which covers Microsoft 365 (Teams, SharePoint, Outlook, OneDrive). Ask several **narrow, single-topic** questions (one each for: conventions; rollout/release practice; recurring gotchas; incidents/regressions; retros; ADRs; design discussions) rather than one broad multi-part question — broad questions time out. **Retry a timeout up to 3×**, and if it still fails, **split it into smaller questions**; record any query that never succeeded in your coverage note. **Bound your own effort:** a connector procedure must not churn indefinitely — cap total questions + retries at a sane budget (on the order of ~20–30 calls), and when you hit it **write up what you have to your candidate file and coverage note and stop**, recording what was left unasked, rather than looping until the orchestrator has to cut you off. When WorkIQ cites a concrete doc URL, read it the right way: an **M365 URL** (SharePoint/OneDrive/Teams) should be read **via WorkIQ itself**, passing the URL to `workiq-ask`'s `fileUrls`; a **public/exported** URL (a published `.docx`/`.pdf`/link) is parsed with the doc-parser (`markitdown`) per the design-docs procedure (item 12). **If neither path can read a cited URL, do not stall — record it in your coverage note** (per the secondary-tool rule above) so the orchestrator can follow up via the design-docs stream (or note it `no-access` if the URL itself is auth-gated), and use WorkIQ's summary meanwhile. **Slack procedure.** Slack is high-value for incidents, reverts, and rollout decisions, but the tooling has sharp edges — pay careful attention to the following:

- **Use `slack_search_public_and_private`** (not just `slack_search_public`) so private channels you're a member of are included. Run **multiple** keyword queries (e.g. `revert`, `regression`, `rollback`, `incident`, `broke`, `<repo name>`, `<service name>`, deploy/rollout terms).
- **The `in:` modifier works** — scope a query with `in:<channel-name>` (no `#`) or `in:<#C0123ABCD>` once you know all relevant channels. Use it to drill into a specific channel's hits.
- **Read decision _threads_, and preserve the timestamps that requires.** Decisions live in replies, so the highest-value move is `slack_read_thread` on a hit's thread, anchored by its **channel id + message timestamp**; permalinks are likewise reconstructed from `(channel, timestamp)`. So **do not read in a mode that strips those identifiers during thread discovery**: a `concise`/preview format that drops each hit's `ts`/permalink leaves you unable to open the thread or cite it. Keep search/list results at a verbosity that **shows each hit's channel id and timestamp**, then read the thread.
- **Avoid giant temp dumps — but not by dropping timestamps.** Full-channel reads can exceed the inline limit and get written to a temp file a subagent may be unable to reopen. Stay in-context with a small `limit` (e.g. 20–30) and **cursor pagination** rather than one huge read. Result `limit` (how many hits) is orthogonal to per-hit verbosity, so you bound size **without** stripping the timestamps the thread/permalink steps need. Prefer **targeted searches + `slack_read_thread`** over dumping an entire channel; never settle for a 500-char preview you then ignore — page through it. **Bound your effort:** cap total search/read calls at a sane budget (order ~20–30) and, when you hit it, write up what you have and stop, recording what was left unread.
- **Discover channels via search, then read by ID.** Channel-name search (`slack_search_channels`) is weak for topic keywords; instead harvest channel IDs from search hits and use `slack_read_channel` by ID. Record the relevant channel IDs you find in your coverage note so they can be reused in later `in:` queries and so a reviewer knows which channels were read.
- **Incident / sensitive channels may require interactive consent.** The Slack skill can refuse to open named incident/security channels in a non-interactive run. Don't block: rely on **keyword search across public+private** (which surfaces incident references and permalinks), and **list the named incident channels you could not open** in your coverage note for a human follow-up.
- **Filter bot noise.** Ops/deploy channels are often dominated by deploy-bot and CI notifications; mine the **human** messages and threads (reverts, locks, "why did this break") and skip the automated pipeline spam, unless it contains useful insights into how users test, deploy, or troubleshoot.
- Record every channel ID read, every query run, and every thread followed in your coverage note, and redact message bodies — memories cite durable artifacts (PRs, files, docs), not raw chat.

**When a source is inaccessible — two distinct cases.** (1) **The connector/tool itself is absent** (the `session_store_sql` / chat / knowledge / warehouse / doc-parser tool was not surfaced to your context): this is a missing connector, not a coverage outcome — follow the connector-source rule above (emit `TOOL_MISSING: <stream-id> <capability>`, write no candidates, do **not** record `no-access`). (2) **The connector is present but a specific resource needs a permission/scope/credential you lack** (e.g. Projects v2 needs `read:project`; the memory export needs staff access; a private Slack channel isn't joined): record it as `no-access` in your coverage note, naming the exact missing access and how to grant it, so the orchestrator can surface it and a future run can be unblocked. Never silently skip either.
