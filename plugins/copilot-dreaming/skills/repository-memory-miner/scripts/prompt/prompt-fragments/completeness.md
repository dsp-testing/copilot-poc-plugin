A thorough pass mines the codebase **to signal-saturation** and reads the artifact history with **explicit breadth targets**. The two have _different_ completeness bars — don't let code "saturation" become an excuse to under-sweep artifacts.

**Code completeness — rigorous saturation, erring toward over-reading.** The bar is: **be highly confident no durable fact was missed.** Enumerate every file in the subsystem and **classify** each as high-signal (entry points, config, public interfaces/exports, types, non-trivial logic, error paths, anything with branching/invariants, in-dir README/docs) or trivial leaf (pure presentational component, generated file, fixture). **Read every high-signal file**, and **open even leaf files briefly to confirm they're actually trivial** rather than assuming. Use the churn-led hot-file hints (session edit-counts are a supporting input) to **order** the reading, not to _limit_ it. A subsystem is `done` only when every high-signal file has been read AND a final sweep surfaces no new durable facts. Prefer over-reading: when unsure whether a file holds signal, read it. Record the basis (files classified, high-signal read, what was skipped as trivial and why).

**Test/fixture/benchmark trees (`TEST` streams) — skim for conventions, not an exhaustive read.** A `TEST` stream is a directory the planner collapsed because it is test, fixture, snapshot, or benchmark material. Mine it **only for durable testing conventions**: the test framework(s) and how tests are run, naming/layout patterns, shared helpers/fixtures and how new tests are expected to reuse them, snapshot/golden update workflows, and any non-obvious rule a contributor must follow. Read a representative sample plus any in-dir README — **not** every case or fixture — and stop once the conventions are captured (a handful of high-value facts is full saturation here). The synthetic inputs and expected-output snapshots themselves are **not** durable signal; do not mine their contents.

**Artifact completeness — exhaustive within the window, hunting subtle signal.** PRs, CI failures, issues, Slack, Discussions, and Projects hold a **long tail of subtle, easily-missed signal** (a one-line reviewer aside, a buried revert reason, a "we tried X and it broke Y" thread). Do **not** sample thinly. Targets: **CI** = read failed logs for **every non-noise failure** in a broad window (exclude only deploy-approval `log not found` runs), and go deep enough into each to find the _root cause_, not just the error line; **PRs** = read **full review threads** (not just top-level comments) across a **large stratified sample** spanning the whole history (recent **and** old, with an emphasis on recent), prioritizing reverted/heavily-discussed/large-churn PRs; **Issues** = all open + a deep closed sample, reading discussion threads; **Slack** = follow every relevant thread to its end, page through cursors, read replies where decisions hide; **Discussions** = full `bodyText` **and all comments** of every entry; **Projects** = item notes/status for in-flight priorities. Read for **implicit** signal (what people assume, argue about, repeatedly clarify), not only explicit statements. **Report the true fraction covered** and, if below target, say so plainly so a follow-up wave can close the gap.
