**Verify against citations** — check every kept memory against its citations and (as applicable) current main-branch state:

- Fact correct but citation **stale/wrong** → emit the corrected record with an updated, verified citation.
- Fact **wrong, or contradicted** by a higher-precedence source → remove it.
- Fact **unverifiable** → the right oracle depends on where the fact's evidence lives, and "I can't confirm it against this checkout" is a valid removal reason only when the checkout is the right oracle:
  - **Repo-local fact** (its evidence should be in the pinned tree but isn't there) → remove it.
  - **Cross-repo / external / operational fact** — one whose evidence inherently lives outside this checkout, so no file here can confirm or deny it: another repo, a live endpoint, or a non-repo artifact (a PR/CI run/release/discussion/doc); cross-repo dependencies are the canonical case. The pinned tree is the **wrong oracle** for it, so do **not** remove it merely because this checkout can't confirm it. Instead verify it against its **actual source** (`gh` for the other repo/artifact, or the live endpoint) and **remove it only if that source contradicts it** (current other-repo state still beats stale PR/chat evidence, per the precedence rule below). If you reach the source but can confirm only part, **narrow the fact to the verified kernel**; if the source is genuinely unreachable **and** nothing contradicts it, **keep it with a brief lower-confidence qualifier in the `fact` itself** (a future agent only ever sees `fact`, so don't present an unverified claim as certain) — a genuine last resort: prefer verifying it or narrowing to a verified kernel, so you emit confirmed content rather than an unverified claim. Don't tag a fact "external" just because you failed to find local evidence: its `fact`/citations must actually name the outside repo, endpoint, or artifact. This carve-out covers **only** the can't-confirm-against-the-tree removal reason — the safety, injection, sanitization, triviality, and mis-scope removal reasons in the curation standards still apply unchanged.
- **Conflict resolution** by source precedence (highest first): (1) current main-branch code/config/docs, (2) current CI workflows, (3) recently merged PRs, (4) issues/discussions, (5) sessions/chat (supporting evidence only). Historical or chat evidence must not override current code without an explicit lower-confidence note.
