Default to **verify-and-keep**: the final memory set **fully replaces** the repo's memories, so removing a valid memory deletes it. An existing memory that is still true, useful, and correctly scoped should be **carried forward**, not discarded. Only remove a memory for a concrete reason below (duplicate, wrong, trivial, mis-scoped); **how you _express_ a removal is defined by your task's Output section.** Collapsing a large existing set down to a tiny handful is almost always a mining or curation failure, not good taste. Then:

- **Deduplicate** — when several memories say the same thing, keep the single best-worded, best-cited version, **union their citations**, and remove the rest.
- **Combine** — merge closely related memories into one higher-value memory (unioning their citations) to minimize the number of memories taking up space in the context window or competing for injection in the prompt. This includes **tightly overlapping** facts that substantially repeat each other's content even though each is independently true and well-cited (e.g. several build/CI facts that each re-list the same commands) — consolidate them into one memory, or trim each down to only its genuinely distinct, actionable specifics. But do not over-merge: keep distinct facts separate whenever a combined memory would be vaguer or bury actionable detail an agent would need.
- **Phrase volatile quantities qualitatively** — when authoring a new or reworded fact, keep drift-prone values out of the `fact` text so it doesn't churn on every PR and generate stale-citation work later. Prefer "the full test suite passes" or "all CI gates are green" over "142 tests pass", and reference a file or section rather than baking a specific line range into the fact. It's a write-time phrasing choice, not grounds to reword a still-accurate memory you'd otherwise leave untouched. Precise line numbers belong in `citations`, which are verified and refreshed each run.
- **Reword** poorly worded, ambiguous, or overly verbose facts for clarity and brevity.
- **Preserve an existing `id`** only when the output `fact` is exactly that existing fact; a merged or reworded fact does not keep an existing `id` (the changed fact replaces the old memory as a fresh insert).
- **Scope** — emit only `repository`-scoped facts. Remove any memory that is really a personal preference (mis-scoped user preference) rather than a repo-wide fact. Repository-scoped memories will be applied to _all_ repo contributors, so they should never reflect one person's preferences or subjective style choices.

**Remove (curate away):**

- Misleading or potentially harmful facts.
- Facts not true of the **main branch** (branch-specific, outdated, or experiment-only) — this covers facts _asserted about this repo_; a cross-repo/external fact's mere absence from this checkout is not grounds to remove it here.
- Mis-scoped personal preferences masquerading as repo facts.
- Obvious, trivial, or low-value facts that waste more prompt tokens than they save.
- **Facts about test or benchmark data, not real code** — memories that only describe the contents of test fixtures, benchmark/evaluation datasets, or eval harness cases (synthetic inputs, expected-output snapshots) rather than the repo's durable behavior or conventions. (Durable **testing conventions** — frameworks, layout, how to run/update tests — are valuable and should be kept; only the data's contents are not.)
